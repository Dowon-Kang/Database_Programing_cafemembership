# 카페 멤버십 회원관리 ERD 설계서

## 1. ERD 작성 기준

- 대상 프로젝트: `jwbook1`의 `ex11` 카페 멤버십 회원관리 기능
- 기준 스키마: `src/main/webapp/ex11/bookmarketdb_3_생성SQL스크립트.sql`
- DBMS 기준: H2 Database
- 모델링 도구: eRWin
- 현재 구현 기준 실제 물리 모델은 `MEMBER` 단일 엔터티 구조임

## 2. 엔터티 목록

### ENTITY 1. MEMBER

- 논리명: 회원
- 물리명: MEMBER
- 설명: 카페 멤버십 회원의 로그인 정보, 기본 인적사항, 스탬프, 관리자 여부를 저장하는 테이블

## 3. 속성 정의

| No | 논리명 | 물리명 | 타입 | 길이 | NULL | KEY | DEFAULT | 설명 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | 회원 아이디 | member_id | VARCHAR | 20 | N | PK | 없음 | 회원 로그인 ID |
| 2 | 비밀번호 | password | VARCHAR | 100 | N |  | 없음 | SHA-256 암호화 비밀번호 |
| 3 | 회원명 | name | VARCHAR | 50 | N |  | 없음 | 회원 이름 |
| 4 | 전화번호 | phone | VARCHAR | 20 | N | UK | 없음 | 회원 연락처, 중복 불가 |
| 5 | 스탬프 개수 | stamp_count | INT |  | Y |  | 0 | 누적 적립 스탬프 |
| 6 | 가입일 | join_date | DATE |  | Y |  | CURRENT_DATE | 회원 가입 날짜 |
| 7 | 관리자 여부 | admin_yn | CHAR | 1 | N |  | 'N' | 관리자 계정 여부, Y/N |

## 4. 키 및 제약조건 정의

### 4.1 기본키

- PK_MEMBER: `member_id`

### 4.2 대체키(유니크)

- UQ_MEMBER_PHONE: `phone`

### 4.3 체크 제약조건

- CK_MEMBER_ADMIN_YN: `admin_yn IN ('Y', 'N')`

### 4.4 기본값

- `stamp_count = 0`
- `join_date = CURRENT_DATE`
- `admin_yn = 'N'`

## 5. 관계 정의

현재 프로젝트의 실제 DB 기준으로는 `MEMBER` 테이블 하나만 사용하므로 물리적인 외래키 관계는 없습니다.

- 관계 수: 0개
- 외래키(FK): 없음
- 자식 엔터티: 없음

즉, eRWin에서는 독립 엔터티 1개로 작성하면 됩니다.

## 6. 업무 규칙

1. 회원은 `member_id`로 식별한다.
2. 전화번호는 중복될 수 없다.
3. 회원은 일반회원과 관리자 계정으로 구분되며 `admin_yn` 값으로 관리한다.
4. 관리자도 동일하게 `MEMBER` 테이블에 저장된다.
5. 스탬프는 회원별 누적값으로 관리한다.
6. 가입일은 회원 등록 시 현재 날짜가 기본 입력된다.
7. 비밀번호는 평문이 아닌 암호화 값으로 저장한다.

## 7. eRWin 입력용 요약

### 엔터티 생성

- Entity Name: `MEMBER`
- Definition: `카페 멤버십 회원 정보 저장`

### 컬럼 입력

1. `member_id` / `VARCHAR(20)` / PK / NOT NULL
2. `password` / `VARCHAR(100)` / NOT NULL
3. `name` / `VARCHAR(50)` / NOT NULL
4. `phone` / `VARCHAR(20)` / NOT NULL / Unique
5. `stamp_count` / `INT` / Default `0`
6. `join_date` / `DATE` / Default `CURRENT_DATE`
7. `admin_yn` / `CHAR(1)` / NOT NULL / Default `'N'`

### Constraint 메모

- Primary Key: `PK_MEMBER(member_id)`
- Unique Key: `UQ_MEMBER_PHONE(phone)`
- Check: `CK_MEMBER_ADMIN_YN(admin_yn IN ('Y','N'))`

## 8. 다이어그램 설명 문장 예시

이 프로젝트의 데이터베이스는 카페 멤버십 회원관리를 위한 단일 엔터티 구조로 설계하였다. `MEMBER` 엔터티는 회원 식별, 로그인, 연락처, 스탬프 적립, 관리자 권한 정보를 함께 관리한다. 회원 아이디를 기본키로 사용하고 전화번호에는 유니크 제약조건을 두어 중복 회원 등록을 방지하였다. 관리자 계정은 별도의 테이블로 분리하지 않고 동일 회원 엔터티 내부의 `admin_yn` 속성으로 구분하였다.

## 9. 제출용 ERD 표기 예시

```text
MEMBER
-------------------------------------------------
PK  member_id     VARCHAR(20)    NOT NULL
    password      VARCHAR(100)   NOT NULL
    name          VARCHAR(50)    NOT NULL
UK  phone         VARCHAR(20)    NOT NULL
    stamp_count   INT            DEFAULT 0
    join_date     DATE           DEFAULT CURRENT_DATE
    admin_yn      CHAR(1)        DEFAULT 'N' NOT NULL
-------------------------------------------------
CHECK (admin_yn IN ('Y', 'N'))
```

## 10. 참고 사항

- 현재 구현 기준에서는 스탬프 적립 이력 테이블이 없으므로 관계형 ERD가 단순하게 보일 수 있습니다.
- 발표나 과제에서 확장 가능성을 설명해야 하면 `STAMP_HISTORY` 같은 이력 엔터티를 향후 확장 모델로 별도 제안할 수 있지만, 현재 프로젝트의 실제 구현 DB 스키마에는 포함되지 않습니다.

## 11. 발표용 설명문

### 11.1 짧은 발표 버전

본 프로젝트의 데이터베이스는 카페 멤버십 회원관리를 중심으로 설계하였습니다. 현재 구현된 물리 모델은 `MEMBER` 단일 엔터티 구조이며, 회원의 로그인 정보와 기본 정보, 스탬프 적립 현황, 관리자 권한을 한 테이블에서 관리하도록 구성하였습니다. 기본키는 `member_id`이고, 전화번호는 중복 회원 등록을 방지하기 위해 유니크 제약조건을 적용하였습니다. 또한 관리자 계정은 별도 테이블로 분리하지 않고 `admin_yn` 속성으로 구분하였습니다.

### 11.2 조금 더 자세한 발표 버전

이 ERD는 현재 구현된 카페 멤버십 시스템의 실제 데이터 구조를 기준으로 작성하였습니다. 회원은 회원 아이디, 비밀번호, 이름, 전화번호, 보유 스탬프, 가입일, 관리자 여부를 속성으로 가지며, 이 정보들은 모두 `MEMBER` 엔터티에 저장됩니다. 비밀번호는 보안을 위해 암호화된 값으로 저장되며, 전화번호는 회원 검색과 적립 처리에 사용되기 때문에 유일해야 합니다. 스탬프는 회원별 누적값으로 관리하고, 관리자 여부는 `Y/N` 체크 제약조건으로 제한하여 데이터 무결성을 확보하였습니다. 현재는 단일 엔터티 구조지만, 향후에는 스탬프 적립 이력 관리를 위해 별도 엔터티로 확장할 수 있습니다.

## 12. 과제 제출용 핵심 정리

### 12.1 엔터티 선정 이유

- 카페 멤버십의 핵심 관리 대상이 회원이므로 `MEMBER` 엔터티를 중심으로 설계함
- 로그인, 회원정보 관리, 스탬프 적립, 관리자 구분을 모두 회원 단위로 처리하므로 단일 엔터티 구조가 현재 요구사항에 적합함

### 12.2 주요 속성 선정 이유

- `member_id`: 회원 식별 및 로그인 처리에 필요
- `password`: 인증 처리에 필요
- `name`: 회원 식별 및 화면 표시용
- `phone`: 회원 검색 및 중복 방지용
- `stamp_count`: 멤버십 적립 기능 구현용
- `join_date`: 가입일 관리용
- `admin_yn`: 관리자와 일반 회원 권한 구분용

### 12.3 무결성 보장 방식

- 엔터티 무결성: `member_id` 기본키 설정
- 도메인 무결성: `admin_yn`은 `Y`, `N`만 허용
- 개체 무결성: `phone` 유니크 제약조건 적용
- 기본값 처리: `stamp_count`, `join_date`, `admin_yn` 기본값 지정

## 13. 확장형 ERD 제안안

아래 내용은 현재 구현된 실제 DB가 아니라, 발표나 과제에서 확장 가능성을 설명할 때 사용할 수 있는 제안 모델입니다.

### 13.1 확장 엔터티: STAMP_HISTORY

- 논리명: 스탬프 적립 이력
- 물리명: STAMP_HISTORY
- 설명: 회원별 스탬프 적립 및 차감 이력을 저장하는 엔터티

| No | 논리명 | 물리명 | 타입 | 길이 | NULL | KEY | 설명 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 1 | 이력 번호 | history_id | BIGINT |  | N | PK | 적립 이력 식별자 |
| 2 | 회원 아이디 | member_id | VARCHAR | 20 | N | FK | MEMBER 참조 |
| 3 | 변경 스탬프 | change_amount | INT |  | N |  | 적립은 양수, 차감은 음수 |
| 4 | 처리 유형 | action_type | VARCHAR | 20 | N |  | 적립, 차감, 사용 등 |
| 5 | 처리 일시 | created_at | TIMESTAMP |  | N |  | 이력 생성 시각 |
| 6 | 비고 | description | VARCHAR | 200 | Y |  | 처리 사유 |

### 13.2 확장 관계

- `MEMBER (1) : (N) STAMP_HISTORY`
- 한 명의 회원은 여러 개의 스탬프 이력을 가질 수 있음
- 하나의 스탬프 이력은 반드시 한 명의 회원에 소속됨

### 13.3 확장형 ERD 설명

현재 시스템은 스탬프 개수를 회원 테이블에 누적값으로만 저장하고 있습니다. 하지만 향후 적립 및 사용 내역 추적이 필요해지면 `STAMP_HISTORY` 엔터티를 추가하여 회원과 1:N 관계로 설계할 수 있습니다. 이렇게 하면 스탬프 증가와 감소 이력을 모두 관리할 수 있고, 적립 통계나 사용 내역 조회 기능으로 확장하기 쉬워집니다.

## 14. eRWin 도식화 가이드

### 현재 구현 기준으로 그릴 때

- 엔터티 1개만 배치: `MEMBER`
- PK와 UK를 속성 앞에 표시
- `admin_yn`에는 체크 제약조건 메모 추가
- 관계선은 그리지 않음

### 확장안까지 함께 제출할 때

- 현재 구현 ERD와 확장 제안 ERD를 분리해서 2장으로 제시
- 1장: 실제 구현 모델 `MEMBER`
- 2장: 확장 모델 `MEMBER` - `STAMP_HISTORY`

## 15. 교수님 질문 대비 포인트

1. 왜 테이블이 1개인가?
현재 구현 범위가 회원관리와 누적 스탬프 관리이기 때문에 가장 단순하고 요구사항에 맞는 구조를 선택했습니다.

2. 왜 관리자 테이블을 따로 두지 않았는가?
관리자도 회원의 한 유형이므로 `admin_yn` 속성으로 구분하는 방식이 현재 요구사항에 더 단순하고 효율적입니다.

3. 왜 스탬프 이력 테이블이 없는가?
현재 시스템은 누적 스탬프 수만 관리하면 충분하기 때문이며, 추후 이력 관리가 필요해지면 `STAMP_HISTORY`로 확장 가능합니다.
